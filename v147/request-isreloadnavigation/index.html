Request.isReloadNavigation

A new read-only boolean property on the Fetch API Request interface that returns true when the navigation was triggered by a user-initiated reload (Refresh button, location.reload(), or history.go(0)). Available in Service Worker FetchEvent handlers to enable cache strategies that differ on reload.

why it exists

Service Workers that serve pages from cache often want to behave differently when the user explicitly reloads — for example, bypassing the cache and fetching fresh resources to match the user's intent. FetchEvent.isReload existed for this purpose but was deprecated because it belongs on the Request object (which is reusable across contexts) rather than on the event. Request.isReloadNavigation provides the same signal in the right place.

Source: blink-dev Intent to Ship — Request.isReloadNavigation attribute, May 2026.

example

// service-worker.js
self.addEventListener('fetch', (event) => {
  const { request } = event;

  // On explicit reload: bypass cache and go network-first
  if (request.isReloadNavigation) {
    event.respondWith(
      fetch(request).then(response => {
        // Optionally update the cache with the fresh response
        const clone = response.clone();
        caches.open('pages-v1').then(cache => cache.put(request, clone));
        return response;
      }).catch(() => caches.match(request))
    );
    return;
  }

  // Normal navigation: serve from cache first
  event.respondWith(
    caches.match(request).then(cached => cached || fetch(request))
  );
});
