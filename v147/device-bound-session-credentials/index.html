Device Bound Session Credentials

Device Bound Session Credentials (DBSC) cryptographically bind a browser session to the device by using the platform's Trusted Platform Module (TPM). Session tokens generated during authentication are tied to a device-bound key pair, so a session cookie stolen by malware cannot be replayed on another machine.

why it exists

Session cookie theft is one of the most prevalent attack vectors on the web. Infostealer malware families such as LummaC2 harvest session cookies from the browser's cookie store and replay them from attacker-controlled machines — bypassing multi-factor authentication entirely because the session is already authenticated.

DBSC addresses this by making the session token cryptographically unusable off the original device. When a user authenticates with a DBSC-enabled site, Chrome generates an asymmetric key pair stored in the device's TPM. The server issues short-lived session tokens that are periodically refreshed via a signed proof-of-possession challenge. If a cookie is exfiltrated, the attacker cannot pass the refresh challenge without access to the TPM-bound private key.

The feature builds on top of existing cookie infrastructure and requires no changes to the Web Platform API for page JavaScript — only the server's session management must implement the DBSC refresh endpoint.

Source: Chrome for Developers — DBSC; chromestatus.com/feature/5140168270413824

how it works

StepDescription
1. LoginUser authenticates normally; server opts in to DBSC via Sec-Session-Registration response header
2. Key generationChrome generates a TPM-backed key pair and registers the public key with the server
3. Short-lived session cookieServer issues a short-lived session cookie (minutes-to-hours lifetime instead of days-to-weeks)
4. Refresh challengeWhen the cookie nears expiry, Chrome presents a signed challenge to the server's refresh endpoint
5. New cookie issuedServer verifies the signature against the registered public key and issues a new short-lived cookie
Theft attemptAttacker reuses the stolen cookie on another machine → refresh challenge fails (no TPM key) → session terminates
Source: WICG DBSC specification

impact on developers

DBSC is transparent to page-level JavaScript. Developers who operate authentication infrastructure must implement the server-side DBSC protocol:

  • Send Sec-Session-Registration header on login response to opt the session into DBSC.
  • Expose a refresh endpoint that validates the signed proof-of-possession JWT Chrome presents.
  • Issue shorter-lived session cookies than normal (Chrome will transparently handle silent refresh).

End users are not affected — the process is invisible to them.
