Respect autocorrect="off" for Windows touch keyboard in TSF

Chrome 148 (Windows desktop) now honours the autocorrect="off" HTML attribute for the Windows touch keyboard. Previously, the Windows Text Services Framework (TSF) would autocorrect typed words even when the focused element had autocorrect="off" set. Ships in Chrome 148 on Windows desktop.

background

The autocorrect HTML attribute (and its IDL equivalent) lets web authors opt editable elements out of autocorrection. For example, a password field, a code editor, or a search box for product SKUs might set autocorrect="off" to prevent the keyboard from mangling user input.

On Windows, Chrome integrates with the operating system's Text Services Framework (TSF) to handle input method interactions including the touch keyboard that appears when using a touchscreen device or tablet. Before Chrome 148, the TSF integration did not propagate the autocorrect="off" attribute to the Windows touch keyboard — so the touch keyboard would apply autocorrections regardless of the attribute's value.

Source: MDN: autocorrect global attribute.

what changed

Chrome 148's TSF integration now detects when the focused editable element has autocorrect="off" and actively reverts any autocorrections applied by the Windows touch keyboard. Concretely:

  • User types "truf" followed by space in an element with autocorrect="off"
  • Before Chrome 148: the touch keyboard replaces "truf" with "true", ignoring the attribute
  • Chrome 148+: "truf " is preserved exactly as typed
<!-- Chrome 148+: Windows touch keyboard no longer autocorrects this field -->
<input type="text" autocorrect="off" placeholder="Enter product code">

<!-- Also works on textarea and contenteditable -->
<textarea autocorrect="off"></textarea>
<div contenteditable autocorrect="off"></div>

scope

This change applies only on Windows, where Chrome uses the TSF API to interface with the touch keyboard. It does not affect:

  • Android (where autocorrect behaviour is controlled differently)
  • macOS or Linux (different input method frameworks)
  • Physical keyboard autocorrect (handled separately)

Note: the autocorrect global HTML attribute itself (with full IDL exposure) ships as a separate feature in Chrome 151. This Chrome 148 change is specifically the TSF-layer fix for Windows touch keyboard behaviour.
