Reuse no-store images on same-src reassignment

Chrome 148 avoids re-fetching an already-decoded image when the same src value is reassigned to an <img> element, even when the image was served with Cache-Control: no-store. This aligns with Firefox and Safari and eliminates unnecessary network round-trips. Ships in Chrome 148 on all platforms.

what changed

When JavaScript assigns img.src = someUrl and the element already has that exact URL as its current source, Chrome previously re-fetched the image from the network if it had been served with Cache-Control: no-store. This is because no-store means "do not store in any cache" — but the image was already decoded and held in the document's in-memory available-image set, not in the HTTP cache.

Chrome 148 changes this: if the image is already available in the same document's in-memory image set with the identical URL, Chrome reuses the decoded image rather than issuing a new network request. Cache-Control: no-store continues to prevent the image from being stored in the HTTP disk cache and from being reused across documents; only the within-document same-src reassignment case is affected.

Source: blink-dev Web-Facing Change PSA msg16010, Chrome 148.

example: image carousel

A common pattern is an image carousel that cycles through a set of images. When cycling back to an image the user has already seen, reassigning the same src previously caused an unnecessary re-fetch for no-store images. Chrome 148 eliminates that re-fetch:

const img = document.querySelector('img');
const srcs = ['/photo-a.jpg', '/photo-b.jpg', '/photo-c.jpg'];
let index = 0;

setInterval(() => {
  index = (index + 1) % srcs.length;
  img.src = srcs[index]; // Chrome 148: reuses if src already decoded in this doc
}, 2000);

Even if the server responds with Cache-Control: no-store, rotating back to /photo-a.jpg will not issue a second network request if the image is still in the document's available-image set.
