Capability Elements: <usermedia> MVP

A new <usermedia> HTML element that provides a browser-controlled button for requesting camera and microphone access — handling both the permission prompt and stream mute/unmute control in a single declarative component, without requiring imperative getUserMedia() calls. Ships in Chrome 150 on desktop and Android.

why it exists

The traditional flow for camera and microphone access requires JavaScript: call navigator.mediaDevices.getUserMedia(), handle the permission prompt asynchronously, then attach the stream to a <video> element. This creates two problems:

  • The permission prompt fires from script execution with no visible user gesture on the element that will show the stream — browsers can block this or warn about it.
  • Users who have previously denied the permission have no obvious recovery path. The site must handle that case and provide UI to re-prompt.

The <usermedia> element solves both by making the element itself the button. Clicking it is an unambiguous user gesture; the browser renders the permission UX inline with the button. After the user grants access, the same element serves as a mute/unmute toggle — no separate UI or JavaScript needed.

Source: WICG/PEPC — usermedia_element.md.

element shape

basic syntax

<usermedia>
  <script type="permissionconstraints">
    {
      "video": true,
      "audio": true
    }
  </script>
</usermedia>

Constraints are supplied as a JSON object inside a child <script type="permissionconstraints">. The presence of "video" or "audio" keys determines whether the element is video-only, audio-only, or audio-video. No separate type attribute is needed.

key attributes and properties

Attribute / PropertyDescription
autostartWhen present, starts stream capture automatically when the element is added to the DOM — no click required.
element.streamA MediaStream property. null while no stream is active. Can be set programmatically to associate an externally-obtained stream.
element.enabledReflects whether the current stream's tracks are enabled (unmuted). Toggled by each click when a stream is active.

events

EventWhen it fires
streamreadyAfter a stream becomes available — either from a successful getUserMedia() call or from setting element.stream programmatically.
userchangedenabledAfter the user clicks the element and toggles the mute state of an active stream.
Source: WICG/PEPC explainer.

two operating modes

mode 1: stream initiation (no stream set)

When the user clicks the element and element.stream is null, the browser calls getUserMedia() with the supplied constraints. This may show the browser's permission prompt. On success, stream is populated and streamready fires.

mode 2: stream control (stream set)

Once a stream is active, each click toggles enabled, muting or unmuting the underlying audio/video tracks. The element's browser-rendered label updates accordingly ("Mute microphone" ↔ "Unmute microphone"). The userchangedenabled event fires after each toggle.

example: camera and microphone button

<usermedia id="capture">
  <script type="permissionconstraints">
    { "video": {"width": {"ideal": 1920}}, "audio": {"sampleRate": 48000} }
  </script>
</usermedia>

<video id="preview" autoplay playsinline></video>

<script>
  const el = document.getElementById("capture");
  const video = document.getElementById("preview");

  el.addEventListener("streamready", () => {
    video.srcObject = el.stream;
  });
</script>

The user sees a browser-styled button. One click triggers the permission prompt (if needed) and starts the stream; subsequent clicks mute and unmute. No JavaScript getUserMedia() call, no catch for denied permissions needed in the basic case.

example: associating an externally-obtained stream

const stream = await navigator.mediaDevices.getUserMedia({ audio: true });

const el = document.querySelector("usermedia");
el.stream = stream;  // element now acts as a mute/unmute toggle for this stream

Setting element.stream directly puts the element immediately into stream-control mode, useful for advanced scenarios like device switching or sharing a stream across multiple elements.

security model

The browser controls the element's rendered content (text, icons, and visual state). Sites cannot override this rendering to disguise the element's purpose — this is what gives the browser confidence that the user's click is an informed intent to grant camera or microphone access. Styling is constrained by the UA for the same reason.

This is the same trust model as the earlier <permission> element experiment; <usermedia> is a refined, media-specific successor that couples the permission request with the stream control lifecycle.
