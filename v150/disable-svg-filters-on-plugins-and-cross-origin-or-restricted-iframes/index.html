Disable SVG filters on plugins and cross-origin or restricted iframes

Chrome 150 prevents SVG CSS filter effects from being applied to cross-origin iframes, sandboxed iframes, and embedded plugins (such as PDFs). When such a filter is encountered, Chrome skips it and uses the nearest ancestor effect that doesn't involve an SVG filter, closing a pixel-readback side-channel.

why it exists

SVG filter primitives such as feBlend, feColorMatrix, and feComposite can be used to read pixel colours from a composited surface. When applied to a cross-origin iframe, these filters can theoretically expose the rendered content of the embedded page to the parent page — a form of cross-origin information leak.

Chrome 150 closes this leak by refusing to apply SVG filter effects to:

  • Cross-origin <iframe> elements
  • Sandboxed <iframe> elements (e.g. those with a sandbox attribute)
  • Embedded plugins (e.g. PDFs rendered via the PDF plugin)

When Chrome encounters such a combination, it traverses the effect tree to find the highest ancestor effect that does not include an SVG filter, and applies that instead. This means non-SVG CSS effects (like opacity, transform, backdrop-filter) applied above or around the restricted frame continue to work normally. Only the SVG-specific filter primitives are blocked.

example

/* This SVG filter WILL be applied — same-origin iframe */
.same-origin-frame {
  filter: url(#blur-effect);
}

/* This SVG filter will be SKIPPED in Chrome 150+ — cross-origin iframe */
.cross-origin-frame {
  filter: url(#custom-svg-filter); /* skipped */
  /* Non-SVG effects still work: */
  opacity: 0.9;
  transform: scale(0.95);
}

/* Workaround: use CSS filter functions (not SVG) for cross-origin frames */
.cross-origin-frame {
  filter: blur(2px) brightness(0.8); /* these still apply */
}
