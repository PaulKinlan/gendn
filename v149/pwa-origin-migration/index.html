PWA Origin Migration

A two-field manifest mechanism that lets an installed Progressive Web App migrate to a new same-site origin — preserving the user's installation — without requiring a manual uninstall and reinstall. Adds migrate_from to the new app's manifest and validates via a .well-known file on the old origin.

Same-site only Migration is restricted to origins that share the same eTLD+1 (e.g. moving from www.example.com to app.example.com). Cross-site migration is not supported. Enterprise force-installed apps cannot be migrated via this mechanism.

why it exists

A Progressive Web App's identity is bound to its web origin at install time. If you rebrand (social.example.com replaces www.example.com/social), reorganise your domain structure, or move a path-scoped app to a subdomain, every existing user is left with a broken or stale installation. Previously they had to manually uninstall the old app and navigate to the new origin to reinstall — a disruptive flow that consistently causes user loss.

PWA Origin Migration defines a two-way handshake: the new app declares which old app it supersedes (via migrate_from in its manifest), and the old origin confirms it consents to the migration (via /.well-known/web-app-origin-association). Chrome validates both, then presents the user with a prompt to migrate their existing installation to the new origin.

Source: WICG/manifest-incubations pwa-migration-explainer.md.

shape of the API

New manifest fields

migrate_from Array. Added to the new app's manifest. Each entry identifies an old app installation that this new app supersedes.
Each entry has:
  • id — the manifest id of the old app (typically its start URL)
  • install_url — a URL on the old origin where Chrome can fetch the old manifest to validate the id
migrate_to Optional. Added to the old app's manifest. Points to the new app's origin. When present, Chrome can discover the migration proactively during a periodic manifest update check — the user doesn't need to visit the new site first.

.well-known verification file

The old origin must host a JSON file at /.well-known/web-app-origin-association that explicitly grants migration permission to the new origin. This prevents a compromised same-site subdomain from silently hijacking an unrelated app.

Migration prompt modes

"suggest" Browser shows a passive notification. The user can dismiss and defer migration indefinitely; the old app continues to work.
"force" Browser shows a blocking dialog at next launch of the old app. The user must either migrate or uninstall. App name and icon updates are deferred until after migration to prevent deceptive branding changes.

What is and is not migrated

InstallationMigrated — the home screen icon, launch shortcut, and OS-level registration move to the new origin.
IndexedDB / Cache StorageNot migrated — origin-bound storage stays on the old origin. Developers must handle their own data migration (e.g. via a service worker on the new origin that reads from the old origin during a transition window).
PermissionsNot migrated initially — the new origin starts with default permissions; users re-grant access as needed.

example

New app manifest (social.example.com)

{
  "name": "Example Social",
  "id": "https://social.example.com/",
  "start_url": "/",
  "scope": "/",
  "migrate_from": [
    {
      "id": "https://www.example.com/social/",
      "install_url": "https://www.example.com/social/manifest.json"
    }
  ]
}

Old .well-known file (www.example.com)

// GET https://www.example.com/.well-known/web-app-origin-association
{
  "https://social.example.com/": {
    "allow_migration": true
  }
}

Old app manifest with proactive migrate_to (optional)

// www.example.com/social/manifest.json
{
  "name": "Example Social",
  "id": "https://www.example.com/social/",
  "start_url": "/social/",
  "migrate_to": "https://social.example.com/"
}

When migrate_to is present, Chrome can discover the migration during its periodic manifest refresh — the user does not need to visit social.example.com first.

Migration flow from the user's perspective

  1. User visits social.example.com (or Chrome detects migrate_to proactively).
  2. Chrome fetches the new manifest and sees migrate_from pointing to the old installed app.
  3. Chrome fetches the old manifest via install_url to validate the id.
  4. Chrome fetches https://www.example.com/.well-known/web-app-origin-association to confirm consent.
  5. If all checks pass, the browser presents a migration prompt.
  6. On user acceptance, the OS-level installation moves to the new origin.

security constraints

Same-site onlyBoth origins must share the same eTLD+1. Moving from example.com to other.com is not allowed.
Two-way handshakeBoth the new manifest (migrate_from) and the old origin's .well-known file must be present. Either alone is insufficient.
id field requiredThe new app's manifest must have an explicit id field. Apps without a stable id cannot participate as the destination of a migration.
Enterprise appsApps managed by the WebAppInstallForceList enterprise policy cannot be migrated via this mechanism.
Source: WICG pwa-migration-explainer.md §"Security".
