Programmatic Scroll Promise

Scroll methods on Element and Window now return a Promise<undefined> that resolves when the scroll animation finishes — or when it is interrupted. After a decade of workarounds, you can finally await element.scrollTo(0, top).

why it exists

Smooth scrolling is async — the browser animates over several frames — but scrollTo() has always returned undefined, leaving developers with no way to know when the animation has finished. Common workarounds included listening to scrollend, starting a requestAnimationFrame loop, or using a setTimeout guess. The scrollend event helps, but it doesn't distinguish between a completed scroll and one that was cut short by a competing scroll. Scroll Promises close that gap: a single await tells you exactly when the scroll is done — or when it was interrupted — without additional event listeners.

Source: promisify-scroll explainer §1 "Motivation".

shape of the API

Every programmatic scroll method on Element and Window now returns Promise<undefined> instead of undefined. Existing callers that ignore the return value are unaffected.

Element

scroll(options) / scroll(x, y) Returns Promise<undefined> that resolves when the scroll settles.
scrollTo(options) / scrollTo(x, y) Alias of scroll(); same Promise semantics.
scrollBy(options) / scrollBy(x, y) Scrolls relative to the current position; returns a Promise.
scrollIntoView(options) Scrolls the element into the viewport; returns a Promise.

Window

scroll(options) / scroll(x, y) Returns Promise<undefined>.
scrollTo(options) / scrollTo(x, y) Same as scroll().
scrollBy(options) / scrollBy(x, y) Returns a Promise.
Source: CSSOM View Module; method signatures from the promisify-scroll explainer.

Promise resolution semantics

Immediate (no-op) The target position is already the current position. Promise resolves synchronously in the next microtask.
Animation complete Smooth-scroll animation finishes without interruption. Promise resolves after the final paint.
Interrupted Another programmatic scroll call or a user gesture starts a new scroll before the current one finishes. Promise resolves (never rejects) with the scroll cut short.

Promises never reject. This preserves backwards compatibility — unhandled rejections would surface silently in existing code that ignores the return value.

Source: promisify-scroll explainer §3 "Interruption behaviour".

example

Sequence two animations

// Scroll to top, then fade the hero in once the scroll settles.
async function revealHero() {
  await window.scrollTo({ top: 0, behavior: "smooth" });
  document.querySelector(".hero").classList.add("visible");
}

Dim a button during scroll

button.addEventListener("click", async () => {
  button.classList.add("loading");
  await container.scrollTo({ left: nextPage * pageWidth, behavior: "smooth" });
  button.classList.remove("loading");
  loadNextPageContent();
});

Handle interruption gracefully

// Promise always resolves — even if the user overrides the scroll.
await element.scrollIntoView({ behavior: "smooth" });
// By here the element is either fully in view or the scroll was
// interrupted. Either way it's safe to act.
element.focus();

Instant scroll (behavior: "instant" / behavior: "auto")

// Instant scrolls also return a Promise — it resolves immediately.
await window.scrollTo({ top: 0, behavior: "instant" });
console.log("at top");
