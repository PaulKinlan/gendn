Resource Timing: spec-compliant Service Worker Router timing fields

Chrome 149 adds four new spec-compliant fields to PerformanceResourceTiming that expose timing and source information for requests handled by the Service Worker Static Routing API — letting developers measure router evaluation latency and verify which source actually served each request. Ships in Chrome 149 on all platforms.

background: Service Worker Static Routing API

The Service Worker Static Routing API (Chrome 116+) allows service workers to declare routing rules in the install event. These rules tell the browser which requests to short-circuit directly to the cache, the network, or the fetch event handler — before waking the service worker JavaScript at all.

Before Chrome 149, PerformanceResourceTiming exposed workerMatchedSourceType and workerFinalSourceType — non-standard attribute names that did not match the W3C Resource Timing specification. Chrome 149 replaces these with the spec-compliant names and adds two timing fields that were missing entirely.

Source: W3C Resource Timing PR #415.

new fields

FieldTypeDescription
workerMatchedRouterSourcestringWhich static router rule was matched. Values: "network", "cache", "fetch-event", "race-network-and-fetch-handler", or "" if no rule matched.
workerFinalRouterSourcestringThe source that ultimately served the request. Same value set. Differs from workerMatchedRouterSource when a fallback occurred (e.g. the matched cache source was cold, so the browser fell back to network).
workerRouterEvaluationStartDOMHighResTimeStampTime at which the browser began evaluating the static routing rules against the request.
workerCacheLookupStartDOMHighResTimeStampTime at which a cache lookup began (only set when a "cache" source was used; 0 otherwise).
Source: WICG service-worker-static-routing-api — resource-timing-api.md.

reading the fields

const observer = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    if (entry.workerMatchedRouterSource) {
      console.log({
        url: entry.name,
        matched: entry.workerMatchedRouterSource,
        final: entry.workerFinalRouterSource,
        routerEvalMs: entry.workerRouterEvaluationStart
          ? entry.workerRouterEvaluationStart - entry.startTime
          : null,
        cacheLookupMs: entry.workerCacheLookupStart
          ? entry.workerCacheLookupStart - entry.startTime
          : null,
      });
    }
  }
});

observer.observe({ type: "resource", buffered: true });

matched vs final source

The matched and final sources can differ when a router rule's primary source is unavailable:

ScenarioMatchedFinal
Static rule sends request to networknetworknetwork
Static rule sends to cache; cache hitcachecache
Static rule sends to cache; cache miss, falls back to networkcachenetwork
No rule matched; fetch event handled by SW JS""fetch-event
No rule matched; no SW fetch handler""""

deprecation of old field names

Chrome previously exposed non-standard names workerMatchedSourceType and workerFinalSourceType. These continue to work in Chrome 149 while usage is monitored, but will be removed once adoption of the spec-compliant names reaches a low enough threshold. Update code to use the new names.
